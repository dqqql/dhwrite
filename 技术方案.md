# 匕首之心团前共创工具技术方案 v1

更新时间：2026-04-29

## 1. 目标

本文档用于固定 MVP 阶段的技术路线、系统边界、同步策略，以及当前“可上线测试”状态下的实现说明。

当前技术目标：

- 支持多人房间协作
- 支持 `free -> co-creation -> normal` 三态流转
- 支持卡牌、疆域、连线、标注、手牌、回合、房主转移的同步
- 优先保证 MVP 可测试，再逐步增强持久化、冲突控制和自动化测试

非目标：

- 不追求逐帧协同拖拽
- 不做离线优先
- 不在 MVP 阶段引入完整账号体系
- 不为超大规模白板场景做过度工程化

## 2. 当前技术栈

- 前端：`React + TypeScript + Vite + Zustand`
- 实时层：`Cloudflare Worker + Durable Objects + WebSocket`
- 当前持久化：`Durable Object storage`
- 预留持久化：`Cloudflare D1`
- 共享协议：`packages/shared`

目录分工：

- `fronted`：前端页面、地图交互、模态框、房间 UI
- `apps/realtime`：Worker、Durable Object、HTTP / WebSocket 接口
- `packages/shared`：共享类型、协议、网格工具、JSON 校验

## 3. 当前实现状态

当前方案已经落地并可运行，处于“可以组织上线测试”的状态。

已打通的主链路：

- 创建房间 / 加入房间
- WebSocket 快照同步与房间更新广播
- 开始共创 / 结束共创
- 抽牌、确认抽牌、打牌、结束回合、强制跳过
- 卡牌移动、缩放、编辑、回收、删除
- 地点疆域独立编辑
- 连线创建、更新、删除
- 标注创建、更新、删除
- 房间备份导出与恢复结构
- 重连恢复

## 4. 系统边界

### 前端负责

- 房间页和所有交互界面
- 地图画布平移、缩放、拖放和本地预览
- 卡牌、连线、标注的可视化渲染
- 本地 UI 状态，如模态框、上下文菜单、连线草稿、标注编辑态

### Worker / Durable Object 负责

- 房间创建、加入和会话签名
- 房间内权威状态串行处理
- 房主转移、在线离线、回合推进
- 卡牌、连线、标注等业务操作的状态更新
- 快照广播
- 导入卡包、导出房间备份

### 共享包负责

- 共享类型：`DhCard`、`MapCard`、`Connection`、`Annotation`、`RoomState`
- WebSocket Client / Server 协议
- `.dhpack.json` / `.dhroom.json` 的结构校验
- 网格、尺寸和疆域归一化工具

## 5. 房间与同步模型

### 房间状态

房间权威状态包含：

- 模式：`free | co-creation | normal`
- 房主、当前回合玩家、回合顺序
- 玩家列表与在线状态
- 手牌、牌堆、地图卡牌、连线、标注
- 房间设置、已选卡包、版本号

### 同步原则

采用“同步业务结果，不同步交互过程”的策略。

立即同步：

- 开始 / 结束共创
- 结束回合 / 强制跳过
- 抽牌与确认抽牌
- 打牌、回收、创建自定义卡
- 新建 / 更新 / 删除连线
- 新建 / 更新 / 删除标注
- 卡牌内容编辑保存

提交式同步：

- 卡牌移动：拖拽结束后提交
- 卡牌缩放：缩放结束后提交
- 地点疆域移动 / 缩放：操作结束后提交

轻量实时状态：

- 在线 / 离线
- 卡牌软锁
- 连接状态提示

## 6. 地图与渲染策略

### 当前方案

MVP 采用自建 DOM 画布：

- 网格背景
- 画布平移 / 缩放
- 地图卡牌绝对定位
- 手牌拖拽上图
- 地点疆域独立渲染
- SVG 连线与标签
- 标注层叠加

这样做的原因：

- 当前实体数量可控
- 对卡牌、疆域、连线、标注的业务定制更直接
- 先验证多人共创链路，再决定是否需要更重的图编辑框架

### 卡牌尺寸

- 不同卡牌类型有默认网格尺寸
- 卡牌支持自由调宽高
- 缩放结果归一化到网格
- 地点卡打出后自动生成疆域，当前仍作为 `MapCard.territory` 挂载

### 卡牌颜色策略

- 类型配置负责提供默认视觉倾向
- 卡牌实例的 `style` 是最终颜色来源
- 因此地图卡、疆域、手牌、抽牌预览、打牌 / 回收动画都以实例 `style` 为准

## 7. 连线与标注策略

### 连线

连线业务数据仅保留：

- `from_card_id`
- `to_card_id`
- `color`
- `label`

标签位置不写入房间状态，而由前端按当前卡牌位置动态计算。

当前渲染约定：

- 连线使用 SVG 二次贝塞尔曲线
- 标签尽量保持“视觉居中”
- 卡牌靠近时标签沿法向适度外移
- 标签单独渲染在上层，避免被卡牌遮挡
- 标签保持可点击，用于进入关系编辑

### 标注

当前标注能力：

- 地图任意位置创建
- 文本编辑
- 拖动位置
- 调整字号
- 删除
- 双端实时同步

当前边界：

- 标注还没有独立软锁
- 多人同时编辑同一标注时，采用最后一次提交生效

## 8. 房间生命周期

### 模式流转

1. `free`
   - 进入房间默认状态
   - 可自由整理地图、标注、连线
2. `co-creation`
   - 房主开始共创后进入
   - 发初始手牌并开启回合
3. `normal`
   - 房主结束共创后进入
   - 回收未打出的手牌，保留地图成果

### 在线与房主转移

- 玩家离线后仍保留在房间列表
- 当前回合遇到离线玩家时可跳过
- 房主离线时自动转移给下一位在线玩家

## 9. 导入导出

### 卡包

- 卡包格式：`.dhpack.json`
- 当前支持整包导入
- 卡包库支持按卡选择导入
- 用户可在测试阶段验证导入后的卡牌是否进入牌堆和卡包库

### 房间备份

- 房间备份格式：`.dhroom.json`
- 包含模式、玩家、手牌、地图卡、连线、标注、设置等完整状态
- 当前支持导出
- 恢复结构已就绪，适合下一阶段继续验证导入恢复

### Markdown 导出

- 当前支持导出叙事摘要 Markdown
- 已包含地图卡、关系和标注内容

## 10. 持久化策略

当前：

- 以 Durable Object storage 为主
- D1 仅保留 schema 和后续升级方向

后续接入 D1 时建议：

- 优先持久化快照而不是操作日志
- 在结构性变更后做防抖写入
- 继续保留基于过期时间的房间清理机制

## 11. 质量与验证

当前验证方式：

- `npm --workspace fronted run build`
- `npm run realtime:typecheck`
- `node smoke-test.mjs`

当前 smoke test 已覆盖：

- 创建房间
- 加入房间
- WebSocket 双端连接
- 开始共创并发初始手牌
- 抽牌三选一
- 打牌到地图
- 锁定并移动卡牌
- 缩放卡牌
- 创建、更新、删除标注
- 创建自定义卡
- 导出 `.dhroom.json`
- 结束共创并回收手牌
- 重连恢复

最新结果：`36 passed, 0 failed`

## 12. 上线测试关注点

建议测试重点：

1. 两到四人同时进入同一房间走完整轮共创。
2. 中途断线重连，确认状态恢复。
3. 多人同时整理地图，观察软锁是否足够清晰。
4. 连线标签和标注在高缩放、低缩放下的可读性。
5. 共创结束后的 `.dhroom.json` 和 Markdown 导出是否满足复盘需求。

## 13. 结论

当前技术路线仍然成立：

- 用 `Cloudflare Worker + Durable Objects` 支撑房间实时协作
- 用 `React + Zustand + 自建 DOM 画布` 快速完成 MVP
- 用 `packages/shared` 保证协议和数据结构一致

当前版本已经达到“可以上线测试”的技术状态，但仍需通过真实多人测试来决定下一轮重点是：

- 继续打磨交互体验
- 加强冲突控制
- 还是补更完整的自动化测试
