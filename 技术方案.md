# 匕首之心团前共创工具技术方案 v1

更新时间：2026-04-29

## 1. 目标

本文档用于锁定 MVP 阶段的技术路线、系统边界、同步策略和关键前端渲染约定。

目标：

- 支撑多人房间协作
- 支撑 `free -> co-creation -> normal` 三态房间流转
- 支撑卡牌、连线、标注、回合、手牌、房主转移的稳定同步
- 优先保证 MVP 可交付，再逐步补强持久化和自动化测试

非目标：

- 不追求逐帧协同拖拽
- 不追求离线优先
- 不在 MVP 阶段引入完整账号体系
- 不提前为超大规模白板场景做过度工程化

## 2. 当前技术栈

- 前端：`React + TypeScript + Vite + Zustand`
- 房间实时层：`Cloudflare Worker + Durable Objects + WebSocket`
- 当前持久化：`Durable Object storage`
- 预留持久化：`Cloudflare D1`
- 共享协议与校验：`packages/shared`

当前目录分工：

- `fronted`：前端页面、画布、弹窗、房间 UI
- `apps/realtime`：Worker、Durable Object、房间 HTTP / WS 接口
- `packages/shared`：TypeScript 类型、协议、网格工具、JSON 校验

## 3. 系统边界

### 前端负责

- 房间页与所有交互界面
- 地图画布平移、缩放、拖放和本地预览
- 由服务端状态派生出的可视化效果
- 本地 UI 状态，如弹窗、上下文菜单、正在连线等

### Worker / Durable Object 负责

- 房间创建、加入、会话签名
- 房间内状态串行处理
- 房主转移、回合推进、软锁控制
- 房间快照广播
- 导入卡包、导出房间备份

### 共享包负责

- `CardType`、`RoomMode`、`DhCard`、`MapCard` 等共享类型
- Client / Server WebSocket 协议
- `.dhpack.json` / `.dhroom.json` 的运行时校验
- 网格和尺寸归一化工具

## 4. 房间与同步模型

### 房间状态

房间核心状态包括：

- 模式：`free | co-creation | normal`
- 房主、当前回合玩家、回合顺序
- 玩家列表与在线状态
- 手牌、牌堆、地图卡片、连线、标注
- 选中卡包、房间设置、快照版本

### 同步原则

采用“同步业务结果，不同步交互过程”的策略。

立即同步：

- 开始 / 结束共创
- 结束回合 / 强制跳过
- 抽牌与确认抽牌
- 打牌、回收、创建自定义卡
- 新建 / 更新 / 删除连线
- 新建 / 删除标注
- 卡片内容编辑保存

提交式同步：

- 卡片移动：拖拽结束后提交
- 卡片缩放：缩放结束后提交
- 地点疆域移动 / 缩放：操作结束后提交

轻量实时状态：

- 在线 / 离线
- 卡片软锁
- 连接状态提示

## 5. 画布与卡牌渲染

### 当前方案

当前 MVP 使用自建 DOM 画布：

- 网格背景
- 画布平移 / 缩放
- 地图卡片绝对定位
- 手牌拖拽上图
- 地点疆域独立渲染
- 连线与标注叠加渲染

这样做的原因：

- 当前实体数量可控
- 地图卡、疆域、手牌、连线交互都更容易按业务定制
- 先验证多人协作闭环，再决定是否迁移到更重的图编辑库

### 卡片尺寸

- 不同卡牌类型有默认网格尺寸
- 卡片支持自由调宽高
- 缩放结果会归一化到网格
- 地点卡打出后自动生成疆域范围，当前仍作为卡片字段挂在 `MapCard` 上

### 卡片颜色策略

这是最近需要明确写入文档的约定：

- 卡牌类型配置只负责提供“类型标签、图标、默认颜色倾向”。
- 卡牌实例自身的 `style` 才是最终渲染颜色来源。
- 因此以下所有区域都应优先使用卡牌实例的 `style`：
  - 地图卡片
  - 地点疆域
  - 手牌
  - 抽牌预览
  - 打牌 / 回收动画

这样可以保证编辑卡片颜色后，所有可见表面都保持一致。

## 6. 连线系统实现约定

### 数据层

连线业务数据只保留：

- `from_card_id`
- `to_card_id`
- `color`
- `label`

不把标签像素坐标写入房间状态。标签位置属于前端派生渲染结果，而不是业务源数据。

### 渲染层

当前连线采用 SVG 渲染，带箭头与颜色分类。

关系标签的前端渲染规则：

- 标签默认应位于连线曲线的中段附近，而不是简单固定 `midY - 常量`
- 标签位置应根据二次贝塞尔曲线的中段点动态计算
- 当两张卡距离较近时，标签应沿曲线法向方向适度外移
- 标签应单独渲染在卡片上层，避免被卡片遮挡
- 标签仍然保留可点击区域，继续作为打开关系编辑器的入口

这一策略可以同时解决“标签不够居中”和“卡片靠近时标签被遮住”的问题。

## 7. 房间生命周期

### 模式流转

当前按三态执行：

1. `free`
   - 玩家入房默认状态
   - 可自由编辑地图、标注、连线、导入导出
2. `co-creation`
   - 房主开始共创后进入
   - 发初始手牌，开启回合
3. `normal`
   - 房主结束共创后进入
   - 回收未打出的手牌，保留地图成果

### 在线与房主转移

- 玩家离线后保留在房间列表中
- 当前回合遇到离线玩家会自动跳过
- 房主离线时自动转交给下一位在线玩家

## 8. 导入导出

### 卡包

- 卡包使用 `.dhpack.json`
- 当前支持整包导入
- 卡包库中支持按卡选择导入
- 卡包类型展示统一用中文标签“地点 / 人物 / 特色”

### 房间备份

- 房间备份使用 `.dhroom.json`
- 包含模式、玩家、手牌、地图卡、连线、标注等完整状态
- 恢复时允许重建牌堆顺序，但要保留地图与手牌结构

## 9. 持久化策略

当前：

- Durable Object storage 为主
- D1 仅保留 schema 和后续升级方向

后续接入 D1 时建议：

- 快照而不是逐操作明细日志优先
- 在结构性变更后做防抖写入
- 房间过期清理仍由 Cron 驱动

## 10. 验证与质量

当前主要验证方式：

- `npm run build`
- `npm run realtime:typecheck`
- `node smoke-test.mjs`

仍需继续补强：

- 前后端自动化测试
- 更细粒度的多人协作实测
- 旧文案与编码问题清理

## 11. 结论

当前路线仍然成立：

- 用 `Cloudflare Worker + Durable Objects` 解决房间实时协作
- 用 `React + Zustand + 自建 DOM 画布` 快速完成 MVP
- 用 `packages/shared` 保证协议和数据结构一致
- 把位置、颜色、标签等“可派生 UI 结果”尽量留在前端计算

最近的文档同步重点有两项：

- 卡片最终显示色以实例 `style` 为准
- 连线关系标签必须保持居中感知，并避免被卡片遮挡
