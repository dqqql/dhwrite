# 匕首之心 · 团前共创工具技术方案 v1

---

## 1. 文档目标

本文档用于在 PRD 已确定的前提下，锁定 MVP 阶段的技术路线、系统边界、同步策略和数据落地方式。

目标：

- 支撑多人房间制协作
- 支撑两模式切换：自由模式 / 共创模式
- 支撑卡牌、连线、标注、回合、手牌的稳定同步
- 控制 Cloudflare 平台上的实时成本与实现复杂度
- 优先保证 MVP 可交付，而不是追求 Figma 级实时白板

非目标：

- 不追求连续拖拽帧同步
- 不追求离线优先
- 不追求复杂账号体系
- 不追求首版就支持移动端完整编辑体验

---

## 2. 总体结论

MVP 采用以下架构：

- 前端：`React + TypeScript + Vite + Tailwind CSS`
- 地图画布：`@xyflow/react`
- 房间实时层：`Cloudflare Worker + Durable Objects + WebSocket`
- 持久化：`Cloudflare D1`
- 静态部署：`Cloudflare Pages`
- 定时清理：`Cloudflare Cron Trigger`
- 共享类型与协议：`zod + TypeScript shared package`

核心原则：

- `Pages` 负责前端站点托管
- `Worker + Durable Objects` 负责实时房间状态与事件协调
- `D1` 负责元数据、房间快照、卡包与导出恢复
- 不同步拖拽过程，只同步拖拽结果

---

## 3. 为什么这样选

### 3.1 为什么不用纯 Pages + D1

这个项目不是普通 CRUD 页面，而是多人房间协作工具。若把所有状态变更都直接写入 D1，会遇到以下问题：

- 地图拖拽、软锁、回合状态都是高频共享状态
- 单个 D1 数据库天然不适合作为实时协作的协调中心
- 高并发下会把数据库当事件总线使用，复杂且成本不优

因此，D1 不承担实时协调职责，只承担持久化职责。

### 3.2 为什么使用 Durable Objects

Durable Objects 很适合本项目的“一个房间一个协调单元”模型：

- 一个房间可映射为一个 Durable Object
- 房间内所有操作在该对象内串行处理，天然避免竞争条件
- 非常适合玩家在线状态、回合顺序、软锁、WebSocket 广播
- 与 Cloudflare 平台部署模型一致

### 3.3 为什么画布优先用 `@xyflow/react`

项目核心对象天然符合“节点 + 边”的数据模型：

- 卡牌是节点
- 连线是边
- 地图具有平移和缩放需求
- 需要自定义节点外观、交互句柄、关系线

`@xyflow/react` 更适合快速完成节点编辑器式 MVP。后续如果出现以下需求，再考虑转向 `react-konva`：

- 大量自由绘制
- 更复杂的图层渲染
- 自定义 Canvas 级交互性能优化

## 4. 部署架构

### 4.1 逻辑结构

```text
Browser
  ├─ 访问 Pages 静态前端
  ├─ 调用 Worker HTTP API
  └─ 连接房间 Durable Object WebSocket

Cloudflare Pages
  └─ 托管 React 前端产物

Cloudflare Worker
  ├─ 房间/卡包/导出 HTTP API
  ├─ 会话签名与邀请码校验
  ├─ 访问 D1
  ├─ 定位并转发到 Durable Object
  └─ Cron 清理过期房间

Durable Objects
  └─ 每个房间一个 DO，维护实时内存状态并广播事件

Cloudflare D1
  ├─ rooms
  ├─ room_snapshots
  ├─ players
  ├─ packs
  ├─ pack_cards
  └─ room_pack_links
```

### 4.2 服务职责

`Pages`

- 托管 SPA
- 负责首屏、静态资源、预览部署

`Worker`

- 创建房间
- 加入房间
- 导入导出
- 创建签名会话
- 落库与恢复快照
- 调用 DO
- 过期房间清理

`Durable Object`

- 房间在线状态
- 回合状态
- 房主转移
- 软锁
- 实时地图事件广播
- 周期性触发快照写入

`D1`

- 卡包与房间元数据
- 持久化快照
- 可恢复状态
- 3 天生命周期管理

## 6. 前端技术栈

### 6.1 核心框架

- `React`
- `TypeScript`
- `Vite`
- `Tailwind CSS`

理由：

- 开发快，生态成熟
- 与 Cloudflare Pages 配合简单
- TypeScript 对共享协议和导出格式很重要

### 6.2 状态管理

推荐：

- 本地 UI 状态：`Zustand`
- 服务器初始化数据：`TanStack Query` 可选

MVP 阶段不建议引入过重的全局状态方案。实时房间状态以 DO 推送为主，前端只维护当前视图状态和客户端派生状态。

### 6.3 画布与地图

使用 `@xyflow/react`，实现：

- 自定义卡牌节点
- 自定义地点范围节点
- 自定义关系边
- 缩放、平移、选中

地图上的“地点范围”可建模为特殊节点字段，而不是额外自由绘制图层，以降低首版复杂度。

### 6.4 UI 组件

建议：

- 基础组件：`Radix UI`
- 样式封装：`shadcn/ui` 风格自建组件

用于：

- 对话框
- 菜单
- 表单
- Toast
- Tabs

自创卡编辑弹窗需要重点打磨，保证简洁美观。

## 7. 实时同步策略

### 7.1 核心策略

采用“事件级同步”，不采用“逐帧同步”。

即：

- 同步结果，不同步过程
- 同步业务事件，不同步所有鼠标移动

### 7.2 同步等级

`立即同步`

- 创建房间后的加入/离开
- 开始共创 / 结束共创
- 回合切换 / 强制跳过
- 房主转移
- 抽牌
- 打牌
- 回收卡牌
- 新建卡牌
- 新建连线 / 删除连线
- 新建标注 / 删除标注
- 内容编辑保存

`轻量实时同步`

- 在线状态
- 正在编辑状态
- 正在拖拽锁定状态

`提交式同步`

- 卡牌拖拽位置变化：拖动结束后一次提交
- 卡牌尺寸变化：缩放结束后一次提交
- 地点范围调整：拖动结束后一次提交
- 文本编辑：点击保存后提交，或使用 500-1000ms 防抖提交

### 7.3 典型交互流程

拖拽卡牌时：

1. 客户端发 `card.lock`
2. DO 广播“该卡正在被某玩家操作”
3. 拖动过程中不广播坐标
4. 玩家松手后发送 `card.move.commit`
5. DO 更新状态并广播最终坐标
6. DO 释放锁

这样其他玩家只会看到：

- 某卡被占用
- 拖拽结束后卡牌跳到新位置

这非常适合本项目的协作强度。

## 8. Durable Object 房间模型

### 8.1 一个房间一个 DO

推荐以 `roomId` 作为 Durable Object 主键。

房间 DO 维护的内存状态包含：

- `roomId`
- `mode`
- `hostPlayerId`
- `currentTurnPlayerId`
- `turnOrder`
- `onlinePlayers`
- `playerHands`
- `mapCards`
- `connections`
- `annotations`
- `locks`
- `selectedPacks`
- `lastSnapshotAt`

### 8.2 DO 职责边界

DO 负责：

- 校验实时事件是否合法
- 串行处理状态变更
- 向房间内连接广播事件
- 生成快照并定时落库

DO 不负责：

- 长期查询分析
- 批量导出复杂格式
- 卡包后台管理

### 8.3 WebSocket 使用方式

客户端加入房间后：

1. 先走 HTTP API 完成邀请码校验与会话初始化
2. 获取短期房间会话凭证
3. 连接 DO WebSocket
4. DO 返回当前完整房间状态

建议事件协议采用 JSON 消息：

```json
{
  "type": "card.move.commit",
  "payload": {
    "cardId": "card_123",
    "x": 120,
    "y": 300
  }
}
```

## 9. D1 持久化策略

### 9.1 D1 的用途

D1 只做“可恢复数据”和“后台元数据”。

包括：

- 房间基本信息
- 邀请码
- 过期时间
- 玩家列表快照
- 房间状态快照
- 卡包与卡牌数据
- 导出记录（可选）

### 9.2 不直接写 D1 的内容

以下高频事件不应逐次直写 D1：

- 鼠标移动
- 拖拽过程坐标
- 临时锁操作
- 每次光标编辑变化

这些由 DO 在内存中管理。

### 9.3 快照时机

建议在以下时机写入 D1：

- 开始共创
- 结束共创
- 回合结束
- 房主转移
- 地图结构性变化后防抖 3-10 秒
- 房间空闲或连接数变为 0

### 9.4 恢复规则

从 `.dhroom.json` 或 D1 快照恢复时：

- 恢复当前模式
- 恢复当前回合玩家
- 恢复回合顺序
- 恢复玩家手牌
- 恢复地图与连线、标注
- 恢复当前房主
- 不要求恢复牌库原始顺序
- 剩余牌可根据当前占用情况重新构建并洗牌

## 10. D1 表设计方向

以下为 MVP 方向，不是最终 SQL 定稿。

### 10.1 `rooms`

字段建议：

- `id`
- `name`
- `invite_code`
- `mode`
- `host_player_id`
- `created_at`
- `expires_at`

### 10.2 `room_snapshots`

字段建议：

- `id`
- `room_id`
- `snapshot_json`

- MVP 可先使用整包 JSON 快照
- 后续若快照体积过大，再拆分结构化表

### 10.3 `players`

字段建议：

- `nickname`

### 10.4 `packs`

字段建议：

- `id`

- `created_at`

其中 `source` 可区分：

- `official`

### 10.5 `pack_cards`

字段建议：

字段建议：

## 11. 会话与身份方案

项目不做注册登录，采用轻量房间会话。

### 11.1 加入房间流程

1. 用户输入邀请码与昵称
2. Worker 校验房间是否存在、是否过期
3. 生成 `playerId`
4. 生成签名会话令牌
5. 前端持有该令牌并用于连接 DO

### 11.2 会话建议

可采用：

- `HttpOnly Cookie`，或
- 本地存储 + 请求头短令牌

MVP 更推荐：

- 连接前由 Worker 下发短时签名 token
- DO 校验签名后允许加入

这样不需要完整账号体系，也能避免任意伪造玩家身份。

## 12. 导入导出策略

### 12.1 卡包导入

支持：

- 导入整个 `.dhpack.json`
- 在卡包查看页导入单张卡牌 JSON

导入时通过 `zod` 校验结构合法性。

### 12.2 房间备份导出

Worker 负责导出 `.dhroom.json`：

- 从 D1 读取最新快照
- 必要时补充实时状态
- 输出符合 PRD 定义的结构

### 12.3 叙事摘要导出

MVP 可先由 Worker 根据地图节点和连线生成 Markdown 文本：

- 地点列表
- NPC 列表
- 特色列表
- 关系列表
- 关键标注列表

## 13. 房间过期与清理

PRD 已确认房间固定保留 3 天。

技术上建议：

- 创建房间时写入 `expires_at = created_at + 3 days`
- 每次活跃操作更新 `last_active_at`
- 但不延长 `expires_at`
- Cron 每天清理一次过期房间

清理内容：

- `rooms`
- `room_snapshots`
- `players`
- `room_pack_links`

若官方卡包为共享资源，则不删除官方卡包表内容。

## 14. 共享协议与校验

建议所有以下数据结构统一放在 `packages/shared` 中：

- 卡牌类型
- 房间状态类型
- WebSocket 消息类型
- `.dhpack.json` schema
- `.dhroom.json` schema

推荐使用 `zod`：

- 前端导入时校验
- Worker API 入参校验
- 导出前结构校验

这能显著降低多人协作项目常见的前后端协议漂移问题。

## 16. 已知风险与后续观察点

### 16.1 风险：`@xyflow/react` 与白板自由度的边界

如果后续发现“地点范围”和“自由布局文本”在 React Flow 中实现成本偏高，需要评估是否：

- 保留节点结构不变，但对特殊元素做额外封装
- 或在后续版本迁移到 `react-konva`

MVP 阶段先不提前复杂化。

### 16.2 风险：快照 JSON 体积变大

若地图内容很多，整包快照会变大。MVP 可先接受，后续再考虑：

- 压缩快照
- 分表存储
- 增量事件日志

### 16.3 风险：WebSocket 连接成本

如果房间数量和在线人数明显上升，需要优先检查：

- 是否使用 DO hibernation 友好的写法
- 是否仍在避免拖拽帧同步
- 是否有多余广播

---

## 17. 最终建议

本项目的 MVP 技术路线建议正式定为：

- `Cloudflare Pages` 托管前端
- `Cloudflare Worker + Durable Objects` 处理实时房间
- `Cloudflare D1` 持久化快照与卡包元数据
- `React + TypeScript + Vite + Tailwind + @xyflow/react` 作为前端主栈
- 采用“事件级同步 + 提交式拖拽同步”的协作策略

这条路线的优点是：

- 与 Cloudflare 部署目标高度一致
- 成本可控
- 实现复杂度适合 MVP
- 为后续扩展更多卡包、更多房间和更复杂协作留下空间
