# 追踪资源房间方案

## 1. 目标

在现有“团前共创房间”之外，新增一种创建时即区分的独立房间类型：

- `共创房间`：保留当前卡牌共创白板逻辑
- `追踪资源`：GM 创建房间，玩家加入后导入角色卡 JSON，在白板中按列展示角色卡，并追踪资源与关键数值

该功能不复用共创模式的卡牌流程，而是复用现有的：

- 大厅创建 / 加入房间
- 房间级实时同步
- 顶栏、设置弹窗、右下角消息区等 UI 语言

## 2. 已确认的产品决策

### 2.1 房间类型

- 创建房间时二选一
- 新房间类型名称定为：`追踪资源`

### 2.2 角色卡导入

- 首版只支持当前仓库内样例 JSON 对应的这一种角色卡格式
- 不做多来源兼容层
- 玩家加入房间后，点击上传 JSON，自动生成自己的角色列

### 2.3 白板展示

- 白板采用“每个玩家一列”的布局
- 每位玩家导入后自动占据一列
- GM 可以调整各列顺序 / 位置
- 普通玩家不能调整其他人的列

### 2.4 编辑规则

- 角色卡默认直接渲染，不进入编辑态
- 所有字段都允许修改
- 详细信息修改需要进入“编辑模式”
- 可追踪资源保持常驻可点击修改，这是核心交互

### 2.5 审批规则

- 房间设置可开启“玩家资源修改需 GM 审批”
- 开启后，仅“玩家发起的资源变更”进入审批流
- GM 的任何修改始终直接生效，不受审批影响

### 2.6 日志与快照

- 右下角消息区记录“数值变更”
- 至少要覆盖：
  - 资源变更
  - 数值字段变更
  - 审批通过后的资源变更结果
- 记录保存在房间快照中

### 2.7 同步策略

- 为降低服务器压力，不做高频白板拖拽同步或编辑过程同步
- 只在“提交修改”时同步给所有人
- 同步时同时写入右下角消息记录

## 3. 房间流程

### 3.1 创建

GM 在大厅中：

1. 输入房间名与昵称
2. 选择房间类型为 `追踪资源`
3. 创建房间并获得邀请码

### 3.2 加入

玩家通过邀请码加入房间后：

1. 先进入空白的追踪房间
2. 点击“上传角色卡”
3. 导入 JSON
4. 系统自动创建该玩家的角色列

限制：

- 每个玩家默认只绑定一张角色卡
- 重新导入角色卡视为“覆盖自己的角色列内容”
- GM 不通过导入创建角色，GM 也可像普通玩家一样导入自己的角色卡

### 3.3 使用

房间内主要操作：

- 玩家修改自己的资源
- 玩家进入编辑模式修改自己的字段
- GM 修改任意玩家的字段和资源
- GM 调整列顺序
- GM 修改顶部恐惧点追踪器
- 审批模式下，GM 处理玩家资源变更申请

## 4. 页面结构

## 4.1 总体布局

沿用当前房间页的视觉语言，但内容切换为资源追踪布局：

- 顶栏：房间名、邀请码、连接状态、导出、设置
- 白板主体：多列角色卡区域
- 顶部固定条：恐惧点追踪器
- 右下角：消息流 / 变更记录

## 4.2 顶部恐惧点追踪器

规则：

- 固定显示在白板最上方
- 上限 12 点
- 所有人可见
- 仅 GM 可修改

交互：

- 点击加减或点选格子修改
- 每次修改都记入右下角消息流

## 4.3 角色列结构

每列对应一名玩家，推荐结构如下：

1. 头部
   - 角色名
   - 玩家昵称
   - 一行摘要信息
2. 基础数值区
   - 等级
   - 闪避
   - 护甲值
   - 重伤阈值
   - 严重阈值
3. 属性区
   - 敏捷
   - 力量
   - 灵巧
   - 本能
   - 风度
   - 知识
4. 资源区
   - 希望点
   - 熟练
   - 生命点
   - 压力点
   - 护甲槽
   - 金币
5. 装备区
   - 护甲文本信息
   - 主武器
   - 副武器
6. 叙事区
   - 种族
   - 职业
   - 社群
   - 子职业
   - 主属性
   - 经历
   - 背景 / 外貌 / 动机

说明：

- `熟练` 虽不在最初口述清单中，但样例 JSON 和参考图中存在，首版建议一并展示
- 若你后续不想保留 `熟练`，可以在开发时作为可裁剪项处理

## 5. 字段映射

首版只面向当前样例 JSON，因此可以按固定映射实现。

### 5.1 基础身份字段

- 角色名字：`name`
- 等级：`level`
- 职业：优先 `professionRef.name`，回退 `profession`
- 种族：
  - 展示为 `ancestry1Ref.name / ancestry2Ref.name`
  - 回退 `ancestry1 / ancestry2`
- 社群：优先 `communityRef.name`，回退 `community`
- 子职业：优先 `subclassRef.name`，回退 `subclass`
- 主属性：
  - 首版建议从六维属性中寻找 `spellcasting = true` 的那一项作为默认值
  - 若找不到，则留空并允许编辑

### 5.2 数值字段

- 闪避：`evasion`
- 护甲值：`armorValue`
- 重伤阈值：`minorThreshold`
- 严重阈值：`majorThreshold`
- 属性：
  - 敏捷：`agility.value`
  - 力量：`strength.value`
  - 灵巧：`finesse.value`
  - 本能：`instinct.value`
  - 风度：`presence.value`
  - 知识：`knowledge.value`

### 5.3 资源字段

- 希望点：`hope` / `hopeMax`
- 熟练：`proficiency`
- 生命点：`hp` / `hpMax`
- 压力点：`stress` / `stressMax`
- 护甲槽：`armorBoxes` / `armorMax`
- 金币：`gold`

### 5.4 装备字段

主武器：

- 名称：`primaryWeaponName`
- 属性/范围：`primaryWeaponTrait`
- 伤害：`primaryWeaponDamage`
- 特性：`primaryWeaponFeature`

副武器：

- 名称：`secondaryWeaponName`
- 属性/范围：`secondaryWeaponTrait`
- 伤害：`secondaryWeaponDamage`
- 特性：`secondaryWeaponFeature`

护甲：

- 护甲名称：`armorName`
- 基础护甲：`armorBaseScore`
- 阈值：`armorThreshold`
- 特性：`armorFeature`

### 5.5 叙事字段

- 经历名称：`experience[]`
- 经历数值：`experienceValues[]`
- 背景：`characterBackground`
- 外貌：`characterAppearance`
- 动机：`characterMotivation`

## 6. 资源交互规则

## 6.1 常驻点击资源

以下资源默认不需要进入编辑模式，直接点选：

- 希望点
- 熟练
- 生命点
- 压力点
- 护甲槽
- 金币
- 恐惧点

建议交互：

- 格子类资源：点击某格即可设定当前值
- 希望点：菱形点选
- 金币：方格点选
- 数字类资源如需要加减按钮，也应在点击后立即提交

## 6.2 编辑模式

以下内容通过编辑弹窗或侧滑面板修改：

- 角色名
- 等级
- 种族 / 职业 / 社群 / 子职业 / 主属性
- 属性值
- 闪避 / 护甲值 / 伤害阈值
- 武器与护甲文本
- 经历
- 背景 / 外貌 / 动机

原则：

- 编辑中不实时同步
- 点击“保存”时一次性提交

## 7. 权限模型

### 7.1 玩家

玩家可以：

- 导入自己的角色卡
- 修改自己的角色卡所有字段
- 修改自己的资源

玩家不能：

- 修改其他人的角色列
- 调整其他人的列顺序
- 修改顶部恐惧点
- 修改房间设置

### 7.2 GM

GM 可以：

- 修改所有人的所有字段
- 修改所有人的资源
- 修改恐惧点
- 调整列顺序
- 开关审批模式
- 审批或拒绝资源申请

## 8. 审批模式

### 8.1 生效范围

仅当房间设置开启后生效：

- 非 GM 玩家对“资源”的修改，需要 GM 审批

不进入审批：

- GM 的任何修改
- 玩家对非资源字段的编辑保存

### 8.2 资源范围

审批模式覆盖的资源建议定义为：

- 希望点
- 熟练
- 生命点
- 压力点
- 护甲槽
- 金币

### 8.3 流程

玩家发起资源修改时：

1. 客户端构造申请
2. 服务端记录为待审批
3. GM 收到申请提示
4. GM 批准后，正式写入资源值并广播
5. 右下角消息流记录批准后的实际变更

建议：

- 玩家本地不要先乐观更新为最终值
- 资源显示“待审批”状态更清晰

## 9. 右下角消息流

### 9.1 记录内容

建议记录以下事件：

- 资源变更
- 数值字段变更
- 恐惧点变更
- 审批通过 / 拒绝
- 玩家导入或覆盖角色卡

### 9.2 文案格式

示例：

- `阿德兰 将希望点从 1 调整为 2`
- `GM 将阿德兰的压力点从 1 调整为 0`
- `GM 将恐惧点从 3 调整为 4`
- `GM 批准了 阿德兰 的护甲槽修改：2 -> 1`

### 9.3 是否记录文本编辑

根据当前确认，消息流重点记录“数值变更”。

因此建议：

- 数值字段和资源字段写入消息流
- 背景、外貌、动机等大段文本编辑不写入消息流

## 10. 同步与性能策略

## 10.1 同步原则

只在提交时广播，不在交互过程广播。

会触发广播的事件：

- 创建追踪资源房间
- 加入房间
- 玩家导入角色卡
- 覆盖角色卡
- 保存编辑模式
- 资源点击修改
- GM 调整列顺序
- GM 修改恐惧点
- GM 修改房间设置
- GM 审批 / 拒绝资源申请

### 10.2 不同步的过程

不做以下高频同步：

- 鼠标移动
- 编辑输入过程
- 资源 hover
- 角色列拖拽中的中间位置

### 10.3 快照内容

`追踪资源` 房间快照应至少包含：

- 房间基础信息
- 房间类型
- 玩家列表
- 恐惧点数值
- 审批设置
- 每位玩家的角色列数据
- 列顺序
- 待审批请求列表
- 右下角消息流
- 版本号与更新时间

## 11. 数据结构建议

建议不要把该功能塞进现有 `co-creation` 房间状态中硬复用，而是显式增加房间类型字段。

### 11.1 创建房间请求

新增字段：

```ts
room_type: 'co-creation' | 'resource-tracker'
```

### 11.2 房间状态建议

推荐结构：

```ts
type RoomType = 'co-creation' | 'resource-tracker'

interface ResourceTrackerRoomState {
  room_type: 'resource-tracker'
  fear: {
    value: number
    max: 12
  }
  settings: {
    imports_enabled: boolean
    resource_change_requires_approval: boolean
  }
  column_order: string[]
  character_columns: ResourceCharacterColumn[]
  pending_resource_requests: ResourceChangeRequest[]
  activity_log: ActivityLogItem[]
}
```

### 11.3 单列数据建议

```ts
interface ResourceCharacterColumn {
  player_id: string
  owner_player_id: string
  imported_at: string
  source_file_name: string
  sheet: {
    identity: ...
    stats: ...
    resources: ...
    equipment: ...
    narrative: ...
    raw_import?: unknown
  }
}
```

说明：

- `raw_import` 建议保留，方便后续排查字段映射问题
- 但前端展示应使用归一化后的 `sheet`

## 12. 前端实现建议

### 12.1 大厅页

新增：

- 创建房间时增加房间类型选择
- 当选择 `追踪资源` 时，提示：
  - GM 创建后玩家可上传角色卡
  - 该房间用于角色资源追踪，不进入共创流程

### 12.2 房间页分流

`RoomPage` 根据 `room_type` 分两套内容：

- `co-creation` -> 现有 `MapCanvas`
- `resource-tracker` -> 新的 `ResourceTrackerBoard`

### 12.3 新组件建议

- `ResourceTrackerBoard.tsx`
- `FearTrackerBar.tsx`
- `CharacterColumn.tsx`
- `CharacterSheetEditModal.tsx`
- `ResourceApprovalPanel.tsx`
- `ActivityFeed.tsx`
- `ImportCharacterModal.tsx`

## 13. 后端实现建议

### 13.1 Durable Object

房间创建时即初始化不同类型状态。

追踪资源房间需新增消息类型：

- `tracker.importCharacter`
- `tracker.updateField`
- `tracker.updateResource`
- `tracker.updateFear`
- `tracker.reorderColumns`
- `tracker.toggleApproval`
- `tracker.approveResourceChange`
- `tracker.rejectResourceChange`

### 13.2 权威写入

所有修改均以房间 DO 为准：

- 客户端只发送意图
- 服务端校验角色归属与权限
- 服务端提交后统一广播

### 13.3 日志写入

服务端在每次成功提交时：

1. 更新权威状态
2. 追加活动日志
3. 广播最新快照或增量事件

## 14. 开发边界

首版建议明确不做：

- 多 JSON 格式兼容
- 同一玩家多角色卡
- 拖拽中实时协同动画
- 复杂白板自由摆放
- 资源修改的批量申请
- 文字评论系统

## 15. 建议开发顺序

### 阶段 1：房间分型与基础状态

- 创建房间增加 `room_type`
- 房间页按类型分流
- 后端初始化 `resource-tracker` 状态

### 阶段 2：角色卡导入与列渲染

- 导入当前 JSON
- 完成字段归一化
- 渲染每位玩家一列

### 阶段 3：核心资源交互

- 希望点 / 生命 / 压力 / 护甲槽 / 金币 / 熟练
- 恐惧点追踪器
- 右下角消息流

### 阶段 4：编辑模式

- 弹窗编辑所有字段
- 保存后提交同步

### 阶段 5：审批模式

- 房间设置开关
- 申请、批准、拒绝
- 待审批提示

### 阶段 6：快照与恢复

- 导出 / 导入追踪资源房间快照
- 恢复日志、列顺序与审批队列

## 16. 当前建议的默认结论

本方案按以下默认值推进：

- `追踪资源` 是独立房间类型，不混入共创模式状态机
- 每位玩家一列，GM 可重排列顺序
- 资源点击即提交，不同步过程态
- 只有玩家资源修改在审批模式下需要 GM 通过
- 所有数值变更进入右下角消息流并写入快照
- 首版严格支持当前这一种 JSON 格式
